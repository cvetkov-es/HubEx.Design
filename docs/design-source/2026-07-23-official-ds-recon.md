# Разведка: перебаза дизайн-системы с Figma на официальную `@hubex/design-system`

Дата: 2026-07-23. Статус: **только разведка, код не менялся.**
Вопрос: стоит ли отвязаться от Figma-макетов и взять источником истины
код основного продукта.

## TL;DR

- «Скопировать код фронтенда» (`HubEx.Frontend.MainApp/src/theme`) — **нет**: это
  замороженный (последний коммит май 2025) легаси Bootstrap-Sass 3, значения которого
  противоречат текущему виду HubEx (радиус `2px` вместо pill, база `15px` вместо `13px`,
  серый primary `#4a4849`, вернувшиеся danger/warning/success-кнопки).
- **Перебазировать источник истины на официальный npm-пакет `@hubex/design-system`** —
  **да, однозначно.** В нём есть структурированный, именованный модуль токенов и типы
  компонентов — ground-truth-версия ровно того, что было вручную реверс-инжинирено из `.fig`.
- Пакет нельзя тянуть в плагины как зависимость (styled-components-рантайм, `react@18`,
  распакованный размер **9.26 МБ**) — поэтому лёгкий `@cvetkov_es/*` (CSS-классы, без рантайма,
  tree-shakeable) сохраняет смысл. Меняется только **источник** значений/API: Figma → пакет.

## Что за пакет

- `@hubex/design-system`, публичный npm, `latest = 0.4.0` (dev-tег `0.2.0`).
  Основное приложение (`HubEx.Frontend.MainApp`) тянет `0.2.0`.
- **Официальная внутренняя DS продуктовой команды** (подтверждено пользователем).
- Мульти-бренд: `themes = { HubEx, MyQRcards, CardEx }`. Нас интересует ветка **HubEx**.
- Экспортирует React-компоненты на styled-components v6, `ThemeProvider`, набор иконок,
  `tokens`-объект. peerDeps: `react@^18.3.1`, `react-dom`, `styled-components@^6.1.11`.

## Токены: diff

Твоя текущая DS: **104** токена (`--hx-*`, в основном цвета + 4 `font/*` из ценза).
Официальная ветка HubEx: **167** листовых значений, полные систематические шкалы.

| Группа | У тебя сейчас | В официальной DS |
|---|---|---|
| Цвета (text/icon/border/background/menu/overlay/avatar) | есть (частично) | 84 значения, полный набор |
| `borderRadius` | pill `9999` (восстановлен) | `small 3` / `medium 5` / `circle 50%` / `pill 9999` — **точно совпадает с твоей правкой** |
| `spaces` (spacing) | **не было** (из ценза, «12/32 выдуманы») | полная шкала `spacing-x0..x16` (0,1,2,3,4,6,8,10,12,14,16,18,20,24,28,32,40,48,56,60,64 px), `unit-base 4px` |
| `sizes` (button/icon/loader/input) | **не было** | `size-button-m 32`, `size-icon-x3..x16` (12..64), `size-input-m 32`, loader s/m/l |
| `typography` | **не было** (11/13/16 из ценза) | структурно: head H0–H3 = 24/20/18/16 (w500), body/paragraph 13, tooltip 12, caption 11 |
| `shadows` | **не было** | `shadow-base`, `shadow-colored`, `shadow-actionpanel` |
| `borderStyle` | — | `border-solid`, `border-dashed` |

Ключевые подтверждения из официального источника (снимают «CANT-TELL» из памяти):
- Радиусы `small=3 / medium=5 / pill=9999` — **ровно** как ты восстановил (против выдуманных 4/8 из 0.1.0).
- Реальная spacing-шкала существует — твои «выдуманные 12/32» больше не нужны.
- Типографика существует переменными — не нужно гадать по цензу.
- **Опечатка `color-backgroundg-warning/-error` присутствует и в официальном пакете** —
  подтверждает твоё решение чинить её у себя, оставляя источник помеченным.

Значения цветов совпадают с твоими исправленными 0.2.0/0.3.0: `#ED1940` (error), `#FF991F`
(warning), `#8DC26F` (success — настоящий), `#32B4EB` (accent), `#1F1F1F`, `#EEEEEE`,
`#777777`, `#999999`.

## Компоненты: diff

Официальная DS (22): Avatar, Badge, Button, Checkbox, Dropdown, Icon, Info, Input,
InputBase, Link, Loader, Pagination, Popover, Radio, Search, SegmentedControl, Select,
Text, TextArea, ThemeProvider, Toggle, Tooltip.

- **В обеих:** Avatar, Badge, Button, Checkbox, Icon, Input, Pagination, Radio, Select,
  Toggle, Tooltip.
- **Только в официальной (у тебя нет):** Dropdown, Info, InputBase, Link, Loader, Popover,
  Search, SegmentedControl, Text, TextArea.
- **Только у тебя (в официальной нет):** Field, Chip, Tag, Alert, Modal, Label, Calendar,
  Menu, Drawer, DatePicker, Tabs, Table.
  → Это твои надстройки из Figma. При перебазе они **не теряются**, но остаются
    неофициальными (без бэкинга источника) — надо явно это отметить.

Пример расхождения API (Button):
- Официальный `ButtonProps`: `variant: primary|secondary|ghost|dashed` (совпадает),
  плюс `size: 'small'|'medium'`, `round`, `loading`, `error`, `fullWidth`, `iconColor`
  (enum токенов), `icon`/`endIcon` как render-функции SVG.
- Твой Button: `variant` совпадает, но `size: 'sm'|'md'` (иное именование), остального нет.
  → При выравнивании API совпадёт с реальным продуктом (важно для ИИ-генерации плагинов).

## Извлекаемость (проверено)

`tokens` лежит в бандле как чистый объектный литерал (`var M={HubEx:{...}}`), извлекается
`eval`-ом в JSON без запуска styled-components. Значит **регенерация токенов из пакета
скриптуема и повторяема** на каждый bump DS — вместо ручного декода `.fig`. Полные типы
компонентов — в `dist/**/**.d.ts`. Извлечённые артефакты разведки:
`/tmp/.../official-hubex.json`, `official-hubex-flat.json`.

## Отдельный пакет «чисто для плагинов»?

Идея здравая, но это в первую очередь вопрос **позиционирования**, а не второй кодовой базы:

- Твой `@cvetkov_es/*` уже и есть «плагинная дистрибуция». Заводить *вторую* систему
  параллельно официальной — рост поддержки без выгоды.
- Рекомендуемая рамка: переспозиционировать существующие пакеты как
  **«plugin build of `@hubex/design-system`»** — тот же официальный источник, но
  переизложенный без рантайма (CSS-классы + тонкий React, `sideEffects:false`,
  tree-shakeable). Один источник, две дистрибуции: тяжёлая официальная (продукт) и
  лёгкая плагинная (ИИ/плагины).
- Открытый вопрос для решения пользователем: имя/scope. Варианты — оставить
  `@cvetkov_es/*`; или (если продуктовая команда согласна) публиковать под их эгидой
  как `@hubex/design-system-plugin` / `@hubex/plugin-ui`.

## Рекомендация

1. **Фаза 1 — токены** (низкий риск, высокая отдача): скрипт-экстрактор `tokens` из пакета
   → регенерация `@cvetkov_es/tokens`. Снимает весь класс «выдуманных значений», добавляет
   реальные spacing/sizes/typography/shadows. Аналог уже сделанной 0.2.0-перебазы, но
   источник надёжный и повторяемый.
2. **Фаза 2 — компоненты**: выровнять API/состояния под официальные типы; добавить
   недостающие (SegmentedControl, Popover, Search, Info, InputBase, Link, Loader, Text,
   TextArea, Dropdown); пометить твои надстройки как неофициальные.
3. Figma → вторичная визуальная сверка, не источник значений.
4. Задокументировать связь: «этот пакет выведен из `@hubex/design-system@X.Y.Z`».
