# HubEx UI — единая дизайн-система для плагинов

**Дата:** 2026-07-16
**Статус:** дизайн согласован, готов к планированию реализации

## Проблема

HubEx — FSM-система, к которой множество команд пишут встраиваемые плагины (в основном
React 18/19, иногда Vue). Сейчас каждый плагин верстается заново, поэтому вид расходится.
Нужен способ, чтобы:

1. Все плагины выглядели одинаково — в системном стиле HubEx.
2. Дизайн переиспользовался разными людьми и **обновлялся централизованно**.
3. При разработке плагина (всегда с участием ИИ) можно было **дать ИИ одну ссылку**, и
   он бы генерировал UI, максимально близкий к системному.

## Источники дизайна (что у нас есть)

- **`Образцы/`** — 17 CSS-дампов экранов (Figma «Copy as CSS»). Абсолютно спозиционированы,
  без структуры компонентов. **Роль: источник значений токенов** (цвета, шрифты, размеры).
- **`figma_reference/`** — 4 нативных `.fig`-файла разделов (Заявки, Компании,
  Объекты/Оборудование, Пользователи). Внутри — реальные компоненты с вариантами.
  **Роль: источник структуры компонентов.**

Именованных Color/Text Styles или Variables в файлах нет — семантические имена токенов
задаются вручную при бутстрапе (делает ИИ, человек только визуально сверяет).

### Извлечённая палитра (подтверждена по частоте в дампах)

| Роль | Значение |
|---|---|
| Текст основной | `#1F1F1F` |
| Текст приглушённый | `#777777`, `#999999` |
| Бренд / акцент | `#32B4EB` (+ `#348AEE`, тинт `#E8F7FD`) |
| Ошибка / danger | `#ED1940` |
| Границы / разделители | `#EEEEEE`, `#E8E8E8`, `#DADADA` |
| Фоны | `#FFFFFF`, `#FEFEFE`, `#F8F8F8` |
| Шрифт | Roboto (Regular/Medium/Bold/Light), база 13px, мелкий 11px, заголовки 16px |
| Иконки | шрифт `material` (Material Icons) |

### Инвентарь компонентов (по всем 4 `.fig`, ~30 шт.)

- **Формы:** Button, Button-Circle, Input, Checkbox, Radio, Select, Toggle
- **Данные:** Table (+ TableHead, TableRow, cell), Pagination, фильтры
- **Даты:** Date-Picker, Calendar (CalendarDay, CalendarHead)
- **Навигация/оверлеи:** Tabs/Tab, Menu, Dropdown, ModalBasic/Modal, Tooltip, Drawer,
  Breadcrumbs
- **Индикаторы:** Tag, Chip, Badge (Dot/Tag/Count/Shift/Expanding), Avatar/Avatar-Pin,
  Label, Alert, Icon

Этого достаточно для старта. Недостающее (графики, канбан и т.п.) добавляется позже как
новый компонент + bump версии — без пересборки системы.

## Принятые решения

| Решение | Выбор | Причина |
|---|---|---|
| Форма распространения | Монорепо → npm-пакеты (зависимость) | единственный вариант с настоящим централизованным обновлением (bump версии) |
| Компоненты | Только React (18/19) | Vue-плагинов мало; полные компоненты нельзя писать один раз на оба фреймворка |
| Поддержка Vue | Токены + CSS (без React-компонентов) | тот же вид ценой одного набора токенов |
| Именование токенов | Вручную при бутстрапе (ИИ) | в Figma нет готовых именованных стилей |
| Тип сборки | Разовый бутстрап + скрипт-страховка | токены меняются редко; постоянный пайплайн не нужен |
| Подключение ИИ | Пакет-зависимость + `llms.txt` | гарантия (импорт реальных компонентов) + правила для генерации нового UI |
| Старт распространения | git-установка из тега → позже GitHub Packages + CI | ноль инфраструктуры на старте, плавный рост в пайплайн |

## Архитектура репозитория

Один монорепозиторий `hubex-ui`:

```
hubex-ui/
├─ packages/
│  ├─ tokens/     → @cvetkov_es/tokens   CSS-переменные (--hx-*) + JS/TS + JSON
│  ├─ css/        → @cvetkov_es/css      базовые классы .hx-btn, .hx-input (React И Vue)
│  └─ react/      → @cvetkov_es/react    React-компоненты: Button, Input, Table, Modal, ...
├─ source/
│  ├─ Образцы/          ← CSS-дампы (источник значений токенов)
│  └─ figma_reference/  ← .fig-файлы (источник структуры компонентов)
├─ tools/harvest.mjs    ← скрипт-страховка: сканит source → отчёт «новые значения»
├─ playground/          ← Vite kitchen-sink: все компоненты на одной странице (не публикуется)
├─ llms.txt             ← ГЛАВНЫЙ файл-ссылка для ИИ
└─ AGENTS.md            ← краткая инструкция для ИИ-агентов плагинов
```

**Граф зависимостей пакетов:** `tokens` ← `css` ← `react`. `tokens` ни от чего не зависит.
Обновил токен → пересобрал → bump версии → плагины подтянули.

## Сборка

Оркестрация: **pnpm workspaces** (локальная линковка пакетов) + **Turborepo**
(сборка по графу зависимостей + кэш). Одна команда `pnpm build`.

| Пакет | Инструмент | Вход → выход |
|---|---|---|
| `@cvetkov_es/tokens` | **Style Dictionary** | `tokens.json` → `variables.css` + `tokens.ts` + `tokens.json` |
| `@cvetkov_es/css` | **PostCSS** | стили поверх переменных → `dist/hubex.css` (автопрефикс + минификация) |
| `@cvetkov_es/react` | **tsup** | `src/*.tsx` → `dist/index.js` (ESM) + `dist/index.d.ts` (типы) |

- React — в `peerDependencies` (`>=18`), пакет не тянет свой React → работают и 18, и 19.
- Витрина: `pnpm dev` поднимает Vite playground с hot-reload.

**Публикация и версии:** **Changesets** — `pnpm changeset` описывает изменение
(patch/minor/major), поднимает версии, пишет CHANGELOG, публикует. На старте — вручную;
позже CI публикует сам при пуше тега. Плагины обновляются `pnpm update @cvetkov_es/react`.

**Вне автосборки:** `.fig`-файлы (источник для ИИ при бутстрапе, не вход билда — парсить
Kiwi в билде оверинжиниринг) и `harvest.mjs` (запускается вручную по желанию).

## Пайплайн токенов (страховка, не рутина)

```
source/Образцы/*.css
   │  tools/harvest.mjs  — авто-скрипт: собирает цвета/шрифты/радиусы/тени + частоту
   ▼
raw-inventory.json  (целиком авто)
   │  diff против tokens.json → «появился новый цвет #XXXXXX, N раз, в токенах нет»
   ▼
tokens.json  (семантика — правит ИИ при реальном изменении дизайна)
   │  Style Dictionary
   ▼
@cvetkov_es/tokens → далее css и react
```

Добавили/удалили выгрузку → `pnpm harvest` показывает только новые значения → ИИ решает,
как назвать или что это шум → `pnpm build`. Запускать необязательно.

## Подключение ИИ (ключевой сценарий)

Два механизма в паре:

1. **Пакет-зависимость (гарантия).** Плагин ставит `@cvetkov_es/react`, ИИ импортирует реальные
   компоненты: `import { Button, Table } from '@cvetkov_es/react'`. «Нарисовать не так» нельзя.
2. **`llms.txt` (файл-ссылка).** В корне репо — один документ: список компонентов и когда
   какой применять, таблица токенов, правила (Roboto, база 13px, бренд `#32B4EB`, радиусы,
   отступы), примеры «как надо / как не надо». ИИ читает его и пишет системный UI даже там,
   где готового компонента нет.

При старте плагина в его `CLAUDE.md`/`AGENTS.md` добавляются две строки: «дизайн-система —
`@cvetkov_es/react`; правила — <raw-ссылка на llms.txt>». Дальше любой человек с ИИ получает
системный вид автоматически.

## План бутстрапа (что делаем в реализации)

1. Скелет монорепо (pnpm + turbo + три пустых пакета).
2. `tools/harvest.mjs` → `raw-inventory.json` из `Образцы/`.
3. `tokens.json` — семантические имена поверх извлечённой палитры + Style Dictionary.
4. `@cvetkov_es/css` — базовые классы на переменных.
5. `@cvetkov_es/react` — core-набор компонентов из `.fig` (начать с Button, Input, Select,
   Checkbox, Radio, Table, Modal, Tag, Badge, Tooltip; остальное — итеративно).
6. `playground/` — kitchen-sink для визуальной сверки с образцами.
7. `llms.txt` + `AGENTS.md`.
8. Changesets + README с инструкцией подключения; первый релиз через git-тег.

## Явно вне scope (YAGNI)

- Vue-компоненты (только токены+CSS для Vue).
- Программный парсинг `.fig` в билде.
- CI/CD на старте (git-установка достаточно; CI — когда устаканится).
- Storybook (заменён минимальным Vite-playground).
- Темизация / дарк-мод (пока нет в исходном дизайне; добавляемо позже через токены).
